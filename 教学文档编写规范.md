# 教学文档编写规范

> **适用范围**：本项目所有教学章节文档的撰写标准
> **读者画像**：初级C++工程师，具备本科工科数学基础，目标为机器人学博士

---

## 十一条编写原则

### 0. 内容深度与丰富度（最高优先级）

**教学文档不是技术速查手册。** 禁止用几句话就"讲清楚"一个概念。每个知识点必须：

- **多角度讲解**：从定义、直觉、几何意义、代数推导、工程应用等多个角度展开
- **完整推导过程**：数学推导必须展示每一步，不允许"容易看出"、"显然"、"略"、"读者自行验证"
- **每一步解释为什么**：不是只写 "由A得B"，而是写 "由A得B，因为...（具体原因）"。学生需要知道每一步的动机和依据，而不仅仅是结果
- **新公式必须交代来龙去脉**：引入任何新公式前，必须回答三个问题：
  1. 这个公式要解决什么问题？（动机）
  2. 它是怎么被推导/发现的？（完整推导过程）
  3. 如果没有这个公式，我们会面临什么困难？（必要性）
- **来龙去脉**：讲清楚这个概念是怎么来的、为什么要这样定义、如果不这样做会怎样
- **关联展开**：与前置知识、后续应用、其他领域的类似概念进行横向对比
- **越丰富越好、越拓展越好、越深入越好**：这是本规范的最高指导原则。宁可多写不可少写。一个概念如果能用 100 行讲得更透彻，就不要压缩到 30 行。写作时应当不断自问："还能从哪个角度再展开？还能再深入一层吗？还有哪些相关知识可以拓展？"
- **深入意味着**：不停留在"是什么"，要讲到"为什么是这样"、"如果不是这样会怎样"、"这个设计背后的权衡是什么"、"历史上尝试过哪些替代方案、为什么被淘汰"
- **拓展意味着**：每个知识点都要向外延伸——与其他章节的联系、在实际工程中的变体、学术前沿的最新进展、不同库/框架的不同实现方式
- **目标行数**：每个章节 **1000 行以上**（这是底线，不是上限），每个小节至少 80-150 行实质内容。复杂推导、编程教学的小节应达到 200 行以上。章节越长越好，不设上限

```
❌ 错误示范：
"Ito等距是Ito积分最重要的性质：E[|∫f dW|²] = ∫E[|f|²]dt。"
（一句话带过，学生根本不知道为什么成立）

✅ 正确示范：
"为什么Ito积分有这么好的性质？让我们从最简单的情况开始理解。

**Step 1: 简单过程的情况**
考虑阶梯函数 φ(t) = Σ eᵢ·1_{(tᵢ,tᵢ₊₁]}，其Ito积分定义为：
  ∫φ dW = Σ eᵢ·(W_{tᵢ₊₁} - W_{tᵢ})

计算其二阶矩：
  E[|∫φ dW|²] = E[|Σ eᵢ·ΔWᵢ|²]
               = Σᵢ Σⱼ E[eᵢ eⱼ ΔWᵢ ΔWⱼ]

关键：当 i ≠ j 时，由于 ΔWᵢ 和 ΔWⱼ 独立（布朗运动的独立增量！），
且 eⱼ 是 F_{tⱼ}-可测的（适应性！），而 ΔWⱼ 独立于 F_{tⱼ}：
  E[eᵢ eⱼ ΔWᵢ ΔWⱼ] = E[eᵢ ΔWᵢ]·E[eⱼ ΔWⱼ] = 0

因此交叉项全部消失！只剩 i = j 的对角项：
  = Σ E[eᵢ² · ΔWᵢ²] = Σ E[eᵢ²]·E[ΔWᵢ²] = Σ E[eᵢ²]·Δtᵢ
  = ∫₀ᵀ E[|φ(t)|²] dt

这就是Ito等距！

**为什么这很重要？** 它说明Ito积分是一个从L²(适应过程)到L²(Ω)的等距映射。
等距意味着保范，保范意味着可以做L²扩张（完备化）——
这正是从简单过程推广到一般过程的关键步骤，和B0_B3中从有理数完备化到实数一样的思想。"
```

### 1. 循序渐进

- 知识点之间必须逻辑顺承，不允许跳跃
- 同一知识点从多个角度讲解（定义→直觉→几何→代数→工程）
- 不同知识点之间展开关联（如：SVD 在 PCA/伪逆/低秩逼近中的三种面目）
- 每个小节开头用一句话说清"这一节解决什么问题"

### 2. 图表结合

- 禁止连续超过 20 行纯文本叙述，必须穿插表格、公式块或示意图
- 对比类内容必须用表格（如：EKF vs UKF vs CKF）
- 流程类内容必须用编号步骤或流程图
- 分类类内容必须用层级列表或树状图

### 3. 实践与测试

- 每个小节结尾设置 2-3 道练习题
- 题目要求开放性和深度，不是简单套公式
- 编程方向：给出可运行的代码框架，让学生填关键部分
- 数学方向：要求手推关键步骤，标注"在草稿纸上完成"
- 算法方向：给出输入输出示例，要求实现并验证

### 4. 先动机后理论（动机 → 反面 → 历史 → 理论）

**禁止直接给定义。** 必须按以下顺序展开：

| 步骤 | 内容 | 示例 |
|------|------|------|
| 动机 | 展示一个具体问题 | "两个旋转的'差'怎么算？" |
| 反面 | 如果不用这个方法会怎样 | "直接 q1-q2 → 结果不是有效旋转" |
| 历史 | 这个方法怎么来的（1-2句） | "Sophus Lie 1870s 发现连续变换群..." |
| 理论 | 正式定义与推导 | "定义：李群是..." |

### 5. 代码与公式的两种模式

根据内容性质选择不同展开方式：

| 情境 | 展开方式 | 重点 |
|------|---------|------|
| **理论推导** | 详细 LaTeX 推导，每一步写清楚 | 推导过程本身，不加代码 |
| **理论应用** | 公式 + 对应算法库实现 | 展示库如何实现该公式 |
| **编程教学** | 讲清楚为什么 + 正确/错误代码对比 + 多种实现对比 | 理解代码背后的设计决策 |

**理论推导模式示例**：

```
Rodrigues 公式推导：

Step 1: 矩阵指数展开为 Taylor 级数
  exp(ω̂θ) = I + ω̂θ + (ω̂θ)²/2! + (ω̂θ)³/3! + ...

Step 2: 利用 ω̂³ = -ω̂ 的循环性质
  奇数项: ω̂θ - ω̂³θ³/3! + ... = sin(θ)·ω̂
  偶数项: ω̂²θ²/2! - ω̂⁴θ⁴/4! + ... = (1-cos(θ))·ω̂²

Step 3: 合并
  exp(ω̂θ) = I + sin(θ)·ω̂ + (1-cos(θ))·ω̂²
```

**理论应用模式示例**：

```cpp
// manif 库实现 Rodrigues 公式
#include <manif/SO3.h>

manif::SO3d R = manif::SO3d::Exp(omega * theta);
// 内部实现等价于上述 Rodrigues 公式

// 逆运算：从旋转矩阵提取轴角
manif::SO3Tangentd log_R = R.log();
// log_R.coeffs() 返回 3D 向量 ω·θ
```

**编程教学模式示例**（C++ 为主）：

编程内容必须遵循以下四步：

**Step 1: 先讲为什么要这样写**

```
为什么 Eigen 要求固定大小矩阵使用栈分配？

因为 SLAM 中的 3×3 旋转矩阵和 4×4 变换矩阵在每帧都要计算成百上千次。
如果每次都 new/delete 堆内存，malloc 的开销会成为瓶颈。
Eigen 的 Matrix3d 直接在栈上分配 72 字节（9 个 double），
编译器甚至可以将其放入 SIMD 寄存器，比堆分配快 10-100 倍。

规则：维度在编译期已知 → 用固定大小（Matrix3d）；
     维度在运行期才知道 → 用动态大小（MatrixXd）。
```

**Step 2: 给出正确写法**

```cpp
// ✅ 正确：固定大小，栈分配，编译器可优化
Eigen::Matrix3d R = Eigen::Matrix3d::Identity();
Eigen::Vector3d omega(0.1, 0.2, 0.3);
Eigen::Matrix3d skew;
skew <<     0, -omega.z(),  omega.y(),
     omega.z(),          0, -omega.x(),
    -omega.y(),  omega.x(),          0;
```

**Step 3: 给出错误写法并解释为什么错**

```cpp
// ❌ 错误 1：用动态大小处理已知维度
Eigen::MatrixXd R(3, 3);  // 堆分配，不必要的 malloc
R = Eigen::MatrixXd::Identity(3, 3);
// 问题：每次调用都 malloc/free，在 1kHz 控制循环中会卡顿

// ❌ 错误 2：未初始化就使用
Eigen::Matrix3d R;        // R 的元素是栈上的垃圾值！
R *= 2.0;                 // 垃圾值 × 2 = 更大的垃圾值
// 问题：不会报错，但结果完全错误。Debug 时非常难发现

// ❌ 错误 3：混用 float 和 double
Eigen::Matrix3f R_float;
Eigen::Matrix3d R_double;
R_double = R_float;       // 编译错误！Eigen 禁止隐式类型转换
R_double = R_float.cast<double>();  // ✅ 必须显式 cast
```

**Step 4: 给出不同实现方式的对比**

```cpp
// === 三种方式计算反对称矩阵 skew(ω) ===

// 方式 A：手动赋值（最清晰，推荐初学者）
Eigen::Matrix3d skew_A;
skew_A <<     0, -w.z(),  w.y(),
         w.z(),      0, -w.x(),
        -w.y(),  w.x(),      0;

// 方式 B：manif 库函数（最简洁，推荐工程使用）
Eigen::Matrix3d skew_B = manif::skew(w);

// 方式 C：叉积矩阵性质 skew(ω)·v = ω × v（理解原理）
// 验证：skew_A * v == w.cross(v) 对任意 v 成立

// 对比：
// | 方式 | 可读性 | 性能 | 适用场景 |
// |------|--------|------|---------|
// | A    | ⭐⭐⭐  | 相同 | 教学、理解原理 |
// | B    | ⭐⭐    | 相同 | 工程代码 |
// | C    | ⭐⭐⭐⭐ | 相同 | 验证正确性 |
```

**编程内容的禁止事项**：

- 禁止只给代码不解释为什么这样写
- 禁止只给正确写法不给错误写法（学生需要知道坑在哪里）
- 禁止只用一种实现方式（至少对比 2 种，让学生理解设计取舍）
- 主要使用 C++ 语言，Python 作为辅助验证工具
- 代码必须可编译/可运行，不允许伪代码冒充实际代码

### 6. 常见陷阱与误区提醒专栏

**陷阱和误区是新手学习中最大的时间黑洞。** 必须给出充分的提示，防止新手的思考走偏。每个小节结尾列出 2-5 个初学者最常犯的错误或最常产生的误解。

**每个陷阱必须包含四个要素**：

| 要素 | 说明 |
|------|------|
| 错误描述 | 新手会怎么想/怎么做 |
| 现象/后果 | 这样做会产生什么问题（越具体越好） |
| 根本原因 | 为什么这是错的（从原理层面解释） |
| 正确做法 | 应该怎么做 + 如何检验自己做对了 |

**陷阱分三类，都必须覆盖**：

**A. 编程陷阱**（代码层面）：

```
⚠️ 编程陷阱：用 Eigen::Matrix3d R; 不初始化就使用
   错误做法：声明后直接参与计算
   现象：R 的元素是栈上的垃圾值（可能是 1e-308 或 1e+15），
        输出结果看起来"差不多对"但数值不稳定，debug 极难发现
   根本原因：C++ 的 POD 类型（和 Eigen 固定大小矩阵）在栈上分配时
            不会自动零初始化，这和 Python/Java 的行为完全不同
   正确做法：
     Eigen::Matrix3d R = Eigen::Matrix3d::Identity();  // 显式初始化
   自检方法：在 Debug 模式下启用 -fsanitize=undefined 或
            用 Eigen 的 EIGEN_INITIALIZE_MATRICES_BY_NAN 宏，
            未初始化的矩阵会变成 NaN，立刻暴露问题
```

**B. 概念误区**（理解层面）：

```
💡 概念误区：认为"旋转矩阵的逆 = 转置"是一个巧合
   新手想法："R 恰好满足 R^{-1} = R^T，真方便"
   实际上：这不是巧合，而是正交矩阵的定义性质。
          R ∈ SO(3) 意味着 R^T R = I 且 det(R) = 1。
          R^T R = I 就是说 R 的列向量构成一组标准正交基。
          几何含义：旋转不改变向量长度（保范），所以其逆就是"反着转"，
          而"反着转"恰好是转置（因为列向量和行向量互换 = 基底对换）。
   为什么重要：理解了这个，就知道为什么 SE(3) 的逆不是简单的转置
             （因为 SE(3) 包含平移，平移部分需要 -R^T t）。
   延伸思考：哪些矩阵的逆等于转置？→ 正交矩阵。
            哪些矩阵的逆等于共轭转置？→ 酉矩阵（量子力学中的旋转）。
```

**C. 思维陷阱**（方法论层面）：

```
🧠 思维陷阱：学完 EKF 后认为"所有非线性滤波都是对 EKF 的改进"
   新手想法："UKF 是改进版 EKF，CKF 是改进版 UKF，
            所以越新越好，直接用最新的就行"
   实际上：EKF/UKF/CKF 不是线性递进关系，而是三种不同的逼近策略：
          - EKF：Taylor 展开（局部线性化）→ 对高度非线性系统偏差大
          - UKF：Sigma 点采样（统计线性化）→ 对称分布表现好
          - CKF：球面径向积分（数值积分）→ 高维时比 UKF 更稳定
          每种方法在不同场景下有优势，没有"最好"的。
   正确思维：选择滤波器应该问三个问题：
          1. 非线性程度多大？（弱 → EKF 够用）
          2. 状态维度多高？（>10 → CKF 优于 UKF）
          3. 计算预算多少？（EKF 最快，UKF/CKF 需 2n+1 或 n^2 点）
   自检方法：对同一个问题跑 EKF 和 UKF，看 NEES 一致性检验
           （B5_A1 节），如果 EKF 的 NEES 合格就不需要换
```

**数量要求**：每个小节至少 2 个陷阱/误区，复杂小节（如 Ito 积分、CBF 设计）至少 4 个。三类（编程/概念/思维）尽量都覆盖。

### 7. 难度分级标记

每个知识点标注难度，初学者可按等级选择性学习：

| 标记 | 含义 | 学习建议 |
|------|------|---------|
| ⭐ | 必学 | 不懂这个后面全看不懂 |
| ⭐⭐ | 核心 | 工程中频繁使用 |
| ⭐⭐⭐ | 进阶 | 理解原理即可，不需要手推 |
| ⭐⭐⭐⭐ | 研究级 | 读论文/改算法时才需要 |

使用方式：在小节标题后标注，如 `### 3.2 Rodrigues 公式推导 ⭐⭐`

### 8. 累积项目

每个方向设计一个贯穿多章的持续项目，每学完一章给项目加一个模块：

| 方向 | 累积项目 | 示例进度 |
|------|---------|---------|
| 足式 | 从零构建四足站立控制器 | Ch1 加载模型 → Ch2 正运动学 → Ch3 逆运动学 → Ch4 PD控制 → Ch5 步态 |
| SLAM | 从零构建 Mini-LIO | Ch1 点云读取 → Ch2 体素滤波 → Ch3 ICP → Ch4 因子图 → Ch5 回环 |
| 数学 | 手写矩阵库 | Ch1 向量类 → Ch2 矩阵乘法 → Ch3 LU分解 → Ch4 SVD → Ch5 最小二乘 |

项目代码保存在独立目录，文档中标注"累积项目：本章新增 xxx 模块"。

### 9. 讨论内容纳入文档

在教学过程中，学习者可能会对某些知识点提出疑问、发现错误、要求深入拓展，或展开讨论。**所有有价值的讨论内容必须回写到文档中**，确保知识点无遗漏。

| 场景 | 处理方式 |
|------|---------|
| 学习者发现文档错误 | 立即修正文档，并在相应位置补充"为什么容易搞错"的说明 |
| 学习者提出深入问题 | 将问答内容整理后插入文档相应小节，作为"深入理解"或"延伸讨论"段落 |
| 学习者需要拓展延伸 | 将拓展内容作为新的子节或附注加入文档 |
| 讨论中产生新的陷阱/误区 | 加入对应小节的"常见陷阱"专栏 |
| 讨论中产生新的代码示例 | 加入对应小节的代码对比区域 |

**原则**：文档是"活文档"，应当持续吸收所有有价值的教学互动内容。最终目标是：任何后来的读者，只看文档就能获得等同于"一对一教学+讨论"的完整知识。
